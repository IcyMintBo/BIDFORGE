# BIDFORGE MVP 功能切入点评估

## 1. 当前代码状态判断

当前 BIDFORGE 项目属于“纯前端原型 + mock 数据 + 规则实验资产”阶段。

| 检查项 | 当前状态 | 判断 |
| ---- | ---- | ---- |
| 前端框架 | `package.json` 使用 Vite、React、TypeScript、Tailwind、lucide-react | 已具备前端开发基础 |
| UI 入口 | `src/App.tsx` 组合 `TopNav`、`HeroBanner`、`ProjectPanel`、`ChapterProgressPanel`、`DraftEditor`、`QualityPanel`、`RiskPanel`、`ForgeLab` | 前台工作台界面已成型 |
| 数据来源 | `src/data/mockData.ts` 提供项目、文件、章节、风险、草稿等 mock 数据 | 仍是静态展示，不是真实任务数据 |
| 组件状态 | `src/components/` 已有项目面板、章节进度、草稿编辑、质量、风险、Forge Lab 等组件 | 适合先接“任务状态 + 结果预览”闭环 |
| 后端 / 本地服务 | `package.json` 无 Express、Fastify、Tauri、Electron、Node API 服务依赖；`src/` 内未发现 `fetch`、`axios`、API 调用或文件系统读写代码 | 尚未有后端或本地服务 |
| 真实文件读写 | 当前浏览器前端不能直接读写 `candidate_skills/`、`docs/`、`trials/`、本地 PDF/DOCX 或输出目录 | 尚未有真实文件读写能力 |
| Runner | `docs/` 中明确记录本地 Runner 未启动；代码中未发现 Runner 调用层 | 尚未有 Runner |
| 任务状态管理 | UI 有章节进度展示，但状态来自 `mockData.ts`，没有真实任务创建、running、success、failed 流程 | 尚未有真实任务状态管理 |
| 导出能力 | 代码中未发现 Blob 下载、DOCX 导出、Markdown 导出或合稿逻辑 | 尚未有导出能力 |
| 规则与试跑资产 | `candidate_skills/`、`docs/`、`trials/` 已存在，且已有 Candidate Skill、Trial、Auditor 文件 | 规则实验场已经比较完整 |
| `runs/` 目录 | 当前未发现 `runs/` 目录 | 尚未形成正式运行输出目录 |

补充判断：当前前端 mock 文案和部分组件文本存在编码异常显示，第一阶段实现时应顺手把本次功能相关的展示文案改为正常中文，但不应扩大为全站重构。

## 2. 候选功能评估

| 候选功能 | 用户可见价值 | 实现难度 | 是否依赖后端/本地服务 | 是否适合作为第一步 | 理由 |
| ---- | ------ | ---- | ----------- | --------- | -- |
| 项目 / 章节选择功能 | 中高 | 低 | 否 | 适合作为闭环组成部分 | 已有项目与章节面板，补交互成本低，但单独做只是在静态 UI 上增加选择，闭环不完整。 |
| 任务创建与任务状态面板 | 高 | 中低 | 否，第一阶段可 mock | 适合作为闭环组成部分 | 能让用户看到“点击后正在生成、生成成功、失败提示”，是从静态原型进入工作台的关键感知。 |
| 本地 Runner 雏形 | 高 | 中高 | 是 | 不适合作为第一步单独启动 | 真实 Runner 需要文件读取、Skill 加载、输出目录、日志和错误处理；如果前台没有消费入口，容易先做成后台脚本孤岛。 |
| Markdown 结果预览 | 高 | 低中 | 否 | 适合作为闭环核心 | 现有 `DraftEditor` 已能展示正文，最适合先改造成“生成结果预览”。 |
| 结果文件下载 | 中高 | 低 | 否 | 适合作为闭环收口 | 浏览器可用 Blob 下载 `.md`，不需要后端即可形成可交付感。 |
| Skill 读取与 used_skill_manifest | 中 | 中高 | 真读取依赖本地服务 | 暂不作为第一步 | 对可信流程很重要，但当前纯前端不能直接读取本地目录；可先在 mock 结果中预留 manifest 字段。 |
| 文件导入 / 资料读取 | 高 | 高 | 真实读取和解析依赖本地服务 | 不适合作为第一步 | PDF/DOCX 解析、来源页码、权限、缓存、错误恢复复杂度高，容易拖慢 MVP。 |
| 章节生成按钮 | 高 | 低 | 否，真实生成依赖后端 | 适合作为闭环入口 | 用户最能感知“工具开始工作”的动作，但必须和任务状态、预览、下载绑定。 |
| 审查结果展示 | 中高 | 中 | 否，第一阶段可 mock | 可作为第二步 | 审查展示有价值，但应先有生成结果，再展示审查。 |
| Word 导出 | 高 | 高 | 通常需要后端或专门库 | 不适合作为第一步 | DOCX 样式、目录、段落、编号、图片和格式映射复杂；应在 Markdown 稳定后再做。 |

## 3. 推荐第一个功能闭环

第一个 MVP 闭环建议为：

```text
前台选择章节 → 点击生成精简稿 → runnerApi mock 返回 Markdown → 前台显示 running / success 状态 → Markdown 结果预览 → 下载 Markdown
```

这个闭环的定位是“前台可运行的生成任务雏形”，不是完整 AI Runner。它先把 UI 从静态展示推进到可交互、可反馈、可下载的最小工作台形态，同时为后续真实本地 Runner 留出 API 层。

推荐优先选择 `5.1 建筑设计` 作为默认演示章节，因为当前 `trials/architectural_design_5_1/` 已经有资料提取、类型判断、写作计划、精简稿和审查结果，可作为第一阶段 mock 结果的内容依据。

## 4. 为什么不是先做其他功能

不先做 Word 导出：Word 导出涉及 DOCX 结构、标题层级、编号、样式、页眉页脚、目录和正式交付格式，复杂度明显高于当前前端状态。应先确认 Markdown 结果流能稳定跑通。

不先做完整 AI Runner：真实 Runner 需要读取招标文件、大纲、Candidate Skill、生成 used_skill_manifest、调用模型、写入输出目录和日志。当前前端没有任务消费接口，先做 Runner 会缺少用户可见闭环。

不先做文件上传解析：PDF/DOCX 解析是高风险基础设施，需要页码、条款、编码、错误提示和大文件处理。第一步可先用 mock/内置 Markdown 验证前台流程。

不先做多项目管理：项目管理会引入项目数据库、文件索引、状态隔离和历史记录。当前最重要的是先证明一个章节任务能从按钮走到结果。

不先做 Skill 可视化编辑：Skill 当前仍处 Candidate 验证阶段，且用户已要求暂停 Candidate Skill 优化。可视化编辑会把 MVP 方向拉回规则实验场。

不先做 Production RC：当前文档明确未进入 Production 或 Production RC。MVP 功能开发应服务于本地工作台，不改变 Skill 状态。

不先做 Forge Lab 深度功能：Forge Lab 当前更像规则后台与演化控制台，适合后期展示审计、规则、回归测试；第一步应让普通前台用户看到章节生成结果。

## 5. 第一阶段具体任务拆分

1. 增加前端任务状态数据结构。
   - 定义章节选择、任务状态、错误信息、生成结果、下载文件名等最小类型。
   - 状态至少包括 `idle`、`running`、`success`、`failed`。

2. 增加 `runnerApi` 占位层。
   - 先做前端 mock Promise，不接真实 Runner。
   - 输入为章节 ID、章节标题和生成模式。
   - 输出为 Markdown 正文、任务元信息、模拟 manifest 摘要。

3. 增加章节选择与“生成精简稿”按钮。
   - 复用现有章节列表和草稿区。
   - 用户可以选择 `5.1 建筑设计`，点击后进入 running 状态。

4. 增加 Markdown 结果预览。
   - 生成成功后，在前台显示 Markdown 结果。
   - 第一阶段不强求完整 Markdown 渲染库，可先用保留换行的预览方式，后续再接解析器。

5. 增加 Markdown 下载。
   - 使用浏览器 Blob 下载 `.md` 文件。
   - 文件名建议包含章节号和时间，例如 `5.1-建筑设计-compact-draft.md`。

## 6. 第一阶段完成标准

第一阶段完成应以用户可见结果为准：

1. 用户能在前台选择一个章节，至少能选择 `5.1 建筑设计`。
2. 用户能点击“生成精简稿”。
3. 页面出现 `running` 状态，并在模拟生成后切换为 `success`。
4. 页面展示生成的 Markdown 内容。
5. 用户能点击按钮下载 Markdown 文件。
6. 生成失败时页面能显示错误状态，而不是静默无反应。
7. 页面文案不再把本次功能相关中文显示成乱码。

## 7. 风险与限制

当前项目是纯浏览器前端，无法直接读写本地文件系统。因此第一阶段不能真正读取 `candidate_skills/`、`docs/`、`trials/`，也不能真正写入 `runs/`。

如果要进入真实本地 AI 工作台，需要后续选择一种本地服务形态：

| 方案 | 适用性 | 说明 |
| ---- | ---- | ---- |
| Express / Node server | 推荐作为下一阶段 | 最轻量，适合提供 `/api/runs`、读取本地文件、写入 Markdown、调用模型。 |
| Electron | 中后期可考虑 | 适合桌面应用和本地文件权限，但会增加打包复杂度。 |
| Tauri | 中后期可考虑 | 更轻量桌面壳，但 Rust/权限桥接会增加工程成本。 |
| 纯前端 | 仅适合第一阶段 | 可做状态、预览、下载；不能做真实本地读写和 Runner。 |

可以先 mock 的部分：

- 章节生成结果；
- 任务状态；
- used_skill_manifest 摘要；
- 审查结果入口；
- 下载 Markdown。

必须后续接真实 Runner 的部分：

- 读取招标文件和投标大纲；
- 读取 Candidate Skill；
- 生成完整 used_skill_manifest；
- 写入 `runs/` 或项目输出目录；
- 记录日志和错误；
- 调用 AI 生成与审查；
- 后续 Word 导出。

## 8. 最终建议

现在最应该先做：

```text
章节选择 + 生成精简稿按钮 + 任务状态 + Markdown 预览 + Markdown 下载
```

暂时不要做：

- 完整 AI Runner；
- Word 导出；
- 文件上传解析；
- 多项目管理；
- Skill 可视化编辑；
- Production RC；
- Forge Lab 深度功能。

如果用户确认开始实现，建议从前端最小闭环开始：先在 `src/` 中增加任务状态类型和 `runnerApi` mock 层，再把 `ChapterProgressPanel` 与 `DraftEditor` 改造成可选择、可生成、可预览、可下载的工作流。该阶段完成后，再启动第二阶段：增加轻量 Node 本地服务，创建 `runs/`，让前台通过 API 调用真实 Runner。
