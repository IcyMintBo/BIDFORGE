## 3. 图纸/方案资料相关性规则

### 证据等级

| 等级 | 证据来源 | 写作权限 | 推荐表达方式 | 示例控制 |
|---|---|---|---|---|
| A 级 | 招标文件、发包人要求、合同条款、明确批复文件 | 可以作为确定事实直接写，优先级最高 | “招标文件明确……” “按发包人要求……” | 项目名称、建设地点、总建筑面积、工期、设计范围、评分点 |
| B 级 | 当前项目图纸、已确认方案、初步设计、地勘、市政接口资料 | 可以写具体技术内容，但应绑定来源 | “结合初步设计/图纸所示……” “根据已确认方案……” | 楼栋层数、平面布局、消防车道、功能分区、管线接口 |
| C 级 | 从 A/B 级资料可以合理推断，但资料未直接写明 | 可以写方向和复核动作，不能写成定论 | “宜结合……复核” “可在施工图阶段进一步校核” | 交通组织优化、机电容量复核、运维空间复核 |
| D 级 | 行业常规经验、通用设计原则 | 只能写原则、方法和风险控制，不得写具体数值、型号、品牌 | “应符合现行规范要求” “应统筹安全、功能和维护” | 多专业协同、质量控制、绿色低碳、设计校审 |
| E 级 | 没有来源、只来自想象、只来自其他项目参考标书 | 禁止写成事实，禁止具体化 | 不写；或写为“待资料确认后确定” | 设备参数、道路宽度、材料品牌、消防系统细节、施工承诺 |

### 禁止凭空具体化的内容

以下内容如果没有 A 级或 B 级证据，BIDFORGE 不得写成确定事实：

- 建筑面积、用地面积、楼栋面积、层数、建筑高度。
- 道路宽度、转弯半径、停车位数量、装卸区尺度。
- 消防系统细节、消防水池容量、泵房参数、喷淋/报警系统配置。
- 设备型号、设备参数、材料品牌、供应商名称。
- 结构体系、基础形式、柱网、荷载、抗震设防参数。
- 机电系统容量、变配电容量、暖通风量、给排水流量。
- 交通流线、物流组织、生产工艺路线的具体安排。
- 甲方未要求的驻场、全过程管理、长期运维、保证通过、保证零缺陷等服务承诺。
- 具体规范条文编号，除非已经核验为当前有效且适用。

### 当前项目的图纸相关性判断

当前 `samples` 中可作为本项目确定事实的主要材料是 `招标文件.pdf`。`设计方案PB(1).pdf` 是其他项目方案，不能作为本项目图纸依据。后续如果要让系统写“结合图纸”的技术内容，需要补充当前项目的以下资料：

- 已确认方案设计文件；
- 初步设计文件及审查资料；
- 总平面图、单体平面/立面/剖面图；
- 市政接口资料；
- 地勘资料；
- 设计任务书或发包人需求任务书。

## 4. 反面教材问题分类

反面教材不能整篇视为“坏文本”。其中有些边界控制语句本身可学习，例如“不作超越招标文件和合同边界的绝对承诺”。但从整体看，两个文件都暴露了后续必须控制的问题：章节过度展开、内容泛化、具体技术内容过多、容易把未核验图纸事实写成定论。

| 问题类型 | 问题说明 | 可能造成的风险 | 审查 Skill 识别方法 | 写作 Skill 避免方式 |
|---|---|---|---|---|
| 泛泛而谈 | 大量使用“系统、统筹、协同、提升、优化”等宏观词，但未绑定招标文件评分点、图纸事实或具体成果 | 得分点响应弱，读起来像通用模板 | 连续段落无项目事实、无来源、无专业对象时标记为空泛 | 每段至少绑定一个项目事实、评分点、专业对象或成果形式 |
| 套话堆叠 | 同一段内反复出现“有利于、提高、减少、保障、形成闭环”等句式 | 文本显长但信息密度低，评审疲劳 | 识别重复句式和同义堆叠 | 限制同类句式连续出现；优先写“措施-依据-成果” |
| 没有项目事实支撑 | 写了很多技术判断，但没有项目名称、建设内容、楼栋、专业范围或来源 | 被认为不针对本项目 | 检查段落是否出现当前项目事实库字段 | 无事实则只写原则；有事实才写项目化表达 |
| 没有响应招标评分点 | 内容很长，但没有覆盖承包人建议书要求的区位、交通、用地现状、周边配套、重难点、解决措施 | 承包人建议书得分低 | 对照评分点建立覆盖矩阵 | 先按评分点建结构，再写正文 |
| 与招标范围关系弱 | 写到与本项目设计范围、施工范围、采购范围无关的内容 | 偏题，甚至暴露复制痕迹 | 检查内容是否可映射至招标范围或发包人要求 | 仅写红线内施工图、专项设计、红线外市政接驳、后续服务等范围内事项 |
| 服务范围扩大 | 写成长期驻场、全过程施工管理、运营管理责任、对施工单位监督管理等 | 可能扩大投标承诺和合同责任 | 识别“管理施工、监督施工、全过程驻场、保证运营”等词 | 使用“后续必要技术配合、设计解释、技术复核”等边界词 |
| 过度承诺 | “确保、必然、彻底、全面消除、保证不发生、一次通过”等绝对表达 | 形成无法兑现承诺，可能引发履约风险 | 扫描绝对化词汇及结果保证句 | 改为“力争、通过……降低风险、满足审查要求、按程序配合” |
| 使用绝对化表达 | 类似“所有、任何、完全、最优、最高、零风险、100%” | 合规和履约风险 | 绝对词命中后要求人工确认 | 除招标文件明确要求外，改为审慎语气 |
| 把施工管理内容写成设计服务内容 | 设计章节中过度写施工组织、材料供应、现场实施、施工顺序控制 | 角色边界混乱，服务范围扩大 | 检查“施工组织、现场管理、采购供应、工人、机械”等词是否出现在设计章节 | 设计章节只写施工图设计、设计配合、专业协同和成果深度 |
| 把其他项目内容迁移进来 | 参考标书中的学校、道路、电力机房、宿迁市民活动中心等内容进入当前项目 | 硬伤，可能被评委直接识别 | 与项目事实库做反向比对，发现外项目地名/功能/单位即高风险 | 参考标书只能提炼表达模式，不复制项目事实 |
| 图纸无依据却写具体技术内容 | 未见图纸就写柱网、荷载、消防车道、设备机房、流线、管线走廊等细节 | 技术事实错误，审查和履约风险 | 对具体数值、系统、空间关系要求来源标记 | 无 B 级证据时写“复核、校核、结合图纸深化” |
| 专业术语看似高级但无法落到本项目 | BIM、CIM、智慧园区、低碳、韧性、全生命周期等词堆叠 | 形成“高级但空”的文本 | 术语后无实施对象或成果时标记 | 术语必须落到图纸、清单、节点、审查资料或校核流程 |
| 前后表述矛盾 | 一处说不扩大范围，另一处写全过程管理或长期运维 | 合同边界不一致 | 跨段检查服务边界词冲突 | 建立统一口径表，所有章节引用同一边界 |
| 章节结构太散 | `02_合理化建议.md` 展开到 1.6.19，`04_优化总则.md` 展开到 2.1.22，粒度过细 | 页面多但重点稀释，阅读成本高 | 统计三级标题数量，超过章节目标即预警 | 第一版章节以评分点和专业边界为主，不无限扩展 |
| 标题很大但正文没有实质内容 | 标题写“机制、体系、路径”，正文仍是原则性描述 | 章节空转 | 检查标题关键词与正文是否有可执行动作 | 每节至少包含“依据、动作、成果、边界” |
| 规范、标准、政策依据疑似过期或无来源 | 参考文本可能引用旧标准或其他地区规范 | 合规风险 | 所有规范条文必须来自招标文件或现行核验清单 | 不主动编规范号；如需引用，先建立规范来源表 |

## 5. 参考标书可学习表达方式

本节只提炼表达模式，不照搬原文。

| 表达模式 | 适用章节 | 表达特点 | 推荐句式或写法 | 禁止误用场景 | 为什么值得学习 |
|---|---|---|---|---|---|
| 响应招标要求 | 项目理解、设计大纲、质量进度措施 | 先点明依据，再展开措施 | “本章节围绕招标文件关于……的要求，重点从……方面进行响应。” | 不得在未读评分点时泛写“全面响应” | 直接对应评分逻辑 |
| 结合项目事实但不乱编 | 工程概述、设计构思、合理化建议 | 用已确认事实开头，后面写控制方向 | “结合招标文件明确的……建设内容，本项目设计应重点关注……” | 不得把参考标书项目事实迁入当前项目 | 增强针对性，同时降低事实错误 |
| 审慎表达 | 合理化建议、优化设计、图纸深化 | 用“宜、可、建议、复核、按程序确认”控制边界 | “建议在施工图设计阶段结合已确认图纸和发包人意见进一步复核……” | 招标文件明确要求的事项不应写得过弱 | 适合处理不确定技术内容 |
| 设计服务边界表达 | 后续服务、施工配合、设计变更 | 明确是设计解释、技术配合，不是施工管理 | “按招标文件约定提供施工图交底、设计变更配合和必要技术服务。” | 不得写“监督施工单位”“全过程管理施工” | 防止扩大服务范围 |
| 工程咨询 / 设计院口径 | 设计质量、专业协同、成果控制 | 强调校审、提资、图纸一致性、审查通过 | “通过专业提资、校审会签和问题闭环，保证设计成果满足审查和实施要求。” | 不得写成施工班组管理口吻 | 更符合设计标文本气质 |
| 合理化建议表达 | 承包人建议书 | 建议内容要有边界、有来源、有落点 | “该建议不改变招标文件确定的建设范围和功能要求，主要用于……阶段进一步复核。” | 不得提出改变功能、降低标准或新增服务责任的建议 | 既能体现主动性，又不乱承诺 |
| 质量保障表达 | 设计质量进度及后期服务 | 组织、流程、校审、成果四层结构 | “从设计输入、专业协同、校审复核、成果交付四个环节控制质量。” | 不得写“保证零错误” | 可执行、可审查 |
| 进度保障表达 | 设计进度、施工配合 | 节点、专业并行、资料前置 | “以设计输入确认、专业提资、阶段校审和成果提交为关键节点组织进度。” | 不得承诺不受外部资料影响 | 稳妥且有管理动作 |
| 限额设计表达 | 优化总则、造价控制 | 强调不降低质量安全功能 | “限额设计应在满足功能、安全、质量和规范要求的前提下开展。” | 不得写为单纯压缩造价 | 与招标文件限额要求一致 |
| 多专业协同表达 | 设计大纲、专项设计 | 强调接口、提资、会审、闭环 | “对建筑、结构、机电、消防、室外工程之间的接口进行统一复核。” | 不得泛写“加强协同”后无动作 | 能把专业复杂性写清楚 |
| 后续服务表达 | 设计后期服务 | 以设计问题处理为边界 | “配合施工图交底、图纸会审、设计变更和审查意见回复。” | 不得写成长期驻场运营管理，除非招标文件明确 | 符合发包人要求 |
| 风险控制表达 | 风险预警、审查报告 | 识别来源、风险、影响、处理动作 | “对资料不一致、接口不清、图纸表达不足等问题形成清单并闭环处理。” | 不得把风险控制写成绝对无风险 | 适合后续审查 Skill |

参考标书中也需要警惕的内容：

- `设计技术标参考(2).docx` 中有大量体系认证、标准年份、企业管理制度，不能直接迁移。
- `03+宿迁市市民活动中心...pdf` 偏施工组织，不适合作为设计服务章节的主要正样本。
- `设计方案PB(1).pdf` 为道路工程，里面的道路宽度、路线名称、区域经济数据不能进入当前项目。
- `排版-技术部分...pdf` 的目录和合规结构可参考，但数据中心机房内容不能进入当前项目。

## 6. BIDFORGE 第一版写作规则草案

1. 招标文件优先原则  
   写作前必须先建立“招标文件事实表”。当参考标书、反面教材、用户口述与招标文件冲突时，以招标文件为准；冲突无法判断时，不写定论，列为需人工确认。

2. 项目事实优先原则  
   每个章节开头应优先使用项目事实库中的信息，如项目名称、建设地点、建设内容、招标范围、评分点。连续两段没有项目事实、专业对象或评分点对应内容时，应判定为泛化风险。

3. 无依据不具体化原则  
   没有招标文件、图纸或明确资料来源时，不得写具体面积、层数、道路宽度、系统容量、设备参数、材料品牌、规范条文和工艺参数。只能写原则、复核动作或待确认事项。

4. 图纸依据分级原则  
   所有技术内容按 A/B/C/D/E 级证据处理。A 级和 B 级可以写具体事实；C 级只能写审慎建议；D 级只能写原则；E 级禁止写。

5. 审慎表达原则  
   对未完全确认的内容，应使用“宜、可、建议、结合……复核、按程序确认、在满足……前提下”等表达。不得使用“确保、彻底、完全、必然、零风险”等绝对化表达。

6. 不扩大服务范围原则  
   设计服务章节只能写施工图设计、专项设计、红线外市政接驳设计、设计变更、审查备案配合、现场技术服务和设计代表驻场等招标文件明确内容。不得写成施工管理、监理、长期运维或替甲方管理施工。

7. 不过度承诺原则  
   凡涉及审查通过、工期、质量、造价、消防、安全、运营效果等结果，应写“通过措施降低风险、满足要求、按程序配合”，不得写“保证一定通过、保证没有问题”。

8. 不写无关项目内容原则  
   写作前必须检测外项目痕迹，包括地名、招标人、工程类型、道路名、学校、机房、图书馆、市民活动中心等。命中后不得进入正式正文。

9. 设计服务与施工管理区分原则  
   设计章节可以写“施工图交底、图纸会审、设计变更、技术复核”；只有施工章节才能写“施工组织、机械设备、劳动力、现场文明施工”。设计章节中出现施工班组、机械安排、现场管理等内容时应预警。

10. 评分点响应原则  
   写作结构必须先映射评分点。承包人建议书至少覆盖区位分析、交通分析、用地现状、周边配套、重难点、解决方案和建议措施。设计大纲至少覆盖项目理解、设计构思、质量进度和后期服务。

11. 专业表达不过度堆砌原则  
   专业术语必须落到对象和成果。例如“管线综合”应说明涉及哪些专业接口、输出什么成果；“BIM”应说明用于碰撞复核还是净高校核。不能只写概念词。

12. 关键判断保留来源原则  
   涉及项目事实、评分要求、服务范围、成果要求、风险判断的句子，应尽量在内部生成记录中保留来源文件和页码。前台正文可不显示来源，但 Forge Lab 必须能追溯。

## 7. BIDFORGE 第一版审查规则草案

| 审查规则 | 审查目标 | 识别方法 | 风险等级 | 修改建议 | 是否可能构成废标风险 |
|---|---|---|---|---|---|
| 招标文件响应检查 | 确认章节是否响应招标文件和评分点 | 与评分点矩阵比对 | 高 | 补充对应评分点内容 | 可能，尤其是响应性评审相关内容 |
| 无来源项目事实检查 | 防止编造项目事实 | 事实库外的面积、地点、单位、规模、工期自动标记 | 高 | 删除或改为待确认表达 | 可能 |
| 技术细节凭空编造检查 | 防止无图纸依据写技术参数 | 识别数值、系统、设备、结构、消防细节是否有来源 | 高 | 改为原则或复核动作 | 可能 |
| 其他项目痕迹检查 | 防止复制参考标书内容 | 检测外项目地名、业主、工程类型、道路名等 | 高 | 删除并替换为当前项目事实 | 可能 |
| 过度承诺检查 | 控制履约风险 | 检测“确保、完全、零风险、必然、100%”等 | 中/高 | 改为审慎表达 | 通常为扣分或履约风险，严重时可能影响响应 |
| 服务范围扩大检查 | 防止超出设计/总承包边界 | 检测长期驻场、管理施工、监督承包商、运营负责等 | 高 | 改为招标文件明确的后续服务 | 可能 |
| 施工管理口径混入检查 | 保持设计章节口径 | 设计章节出现劳动力、机械、班组、施工现场管理等词 | 中 | 移至施工章节或删除 | 一般非废标，但影响专业性 |
| 绝对化表达检查 | 降低不当承诺 | 绝对词列表扫描 | 中 | 改为“在……前提下”“力争”“降低风险” | 一般非废标 |
| 规范过期或无依据检查 | 防止错引标准 | 检查规范号是否来自招标文件或已核验清单 | 高 | 删除规范号或标注待核验 | 可能 |
| 得分点遗漏检查 | 提升评审针对性 | 章节覆盖矩阵显示空项 | 中/高 | 增补对应小节 | 一般为扣分风险 |
| 标题正文不匹配检查 | 防止标题大正文空 | 标题关键词未在正文动作/成果中出现 | 中 | 重写为“依据-动作-成果”结构 | 一般非废标 |
| 空泛套话检查 | 提高信息密度 | 宏观形容词密度高且无项目事实 | 中 | 加入项目事实和具体控制对象 | 一般为扣分风险 |
| 前后矛盾检查 | 保持统一口径 | 服务范围、工期、质量、面积等跨段比对 | 高 | 统一到事实库口径 | 可能 |
| 图纸无依据内容检查 | 防止设计事实错误 | 图纸相关内容无 A/B 级来源 | 高 | 删除、降级为复核建议或补充来源 | 可能 |

## 8. BIDFORGE 规则提取 Skill 草案

Skill 名称：`bidforge-rule-extraction`

目标：  
从招标文件、参考标书、反面教材、用户人工修改稿中提取可复用规则，为写作 Skill 和审查 Skill 提供补丁。

输入材料：

- 招标文件；
- 当前项目图纸/方案/初设/地勘；
- 参考标书；
- 反面教材；
- 模型原始输出；
- 用户人工修改后的版本；
- 用户评价或批注。

输出内容：

- 项目事实增量；
- 正向表达模式；
- 负面问题类型；
- 可泛化规则；
- 本项目特例；
- 写作 Skill 补丁；
- 审查 Skill 补丁；
- 需要人工确认的问题清单。

工作流程：

1. 识别材料角色，不混用招标文件、参考标书、反面教材。
2. 从招标文件抽取 A 级事实和硬约束。
3. 从图纸/方案资料抽取 B 级证据，并保留页码、图名或文件名。
4. 从参考标书提取句式、结构、口径，不提取项目事实。
5. 从反面教材提取问题类型，不把具体错误内容硬编码成规则。
6. 将规则分为“通用规则”“项目规则”“临时规则”“禁止规则”。
7. 输出规则候选，交由 Auditor 审查。

禁止行为：

- 不得把参考标书整段复制为正样本。
- 不得把反面教材整篇视为负样本。
- 不得把某个项目特例推广为通用规则。
- 不得生成正式标书正文。
- 不得无来源补充项目事实。

如何判断一条规则是否可以泛化：

- 规则是否不依赖某个项目名称、地点、甲方、功能类型；
- 是否可在多个设计类标书中执行；
- 是否能被审查器检测；
- 是否有正反样本支持；
- 是否不违反招标文件优先原则。

如何区分“通用规则”和“本项目特例”：

- “无依据不具体化”是通用规则。
- “本项目总建筑面积 91613.67 ㎡”是本项目事实，不是通用规则。
- “重卡智能制造产业园需关注厂房、综合楼、35KV 开闭所、室外道路、海绵城市”是本项目规则，不应迁移到学校或市政道路项目。

如何从反面教材中提取负面规则：

- 找出用户不满意的具体位置；
- 标记问题类型，如泛化、越界、无图纸依据、过度承诺；
- 把具体文字转化为抽象检测条件；
- 给出推荐替代方式。

如何从参考标书中提取正向表达：

- 只抽“表达方式”和“章节组织方式”；
- 去掉项目名称、地名、参数、业主、工程类型；
- 检查是否存在过度承诺；
- 形成可复用句式模板。

## 9. BIDFORGE 写作 Skill 草案

Skill 名称：`bidforge-bid-writing`

目标：  
根据招标文件、项目事实库、图纸证据、参考表达库和章节任务，生成稳妥、合规、项目化的标书正文。

输入材料：

- 章节任务；
- 招标文件事实库；
- 评分点矩阵；
- 图纸/方案证据表；
- 参考表达库；
- 禁止表达清单；
- 用户指定口径。

输出格式：

- 章节标题；
- 正文；
- 内部来源记录；
- 不确定事项清单；
- 建议插图/图位提示，若用户要求。

写作流程：

1. 确认章节属于承包人建议书、设计大纲、实施方案或其他部分。
2. 调取对应评分点和招标文件要求。
3. 调取项目事实库，建立本章节可用事实列表。
4. 判断是否需要图纸依据；没有图纸依据时禁止具体化。
5. 选择参考表达模式，而非复制参考标书。
6. 生成正文前先列出写作边界。
7. 生成正文后交给审查 Skill 扫描。
8. 根据审查结果修改。

必须遵守的规则：

- 招标文件优先；
- 项目事实优先；
- 无依据不具体化；
- 不扩大服务范围；
- 不过度承诺；
- 不写无关项目内容；
- 不混淆设计服务与施工管理；
- 对关键判断保留来源。

禁止行为：

- 编造图纸事实；
- 引入其他项目名称或内容；
- 使用绝对化承诺；
- 写超出招标范围的服务；
- 用页数填充替代有效内容；
- 在没有核验时写规范条文号。

不确定内容表达方式：

- “可结合后续已确认图纸进一步复核……”
- “建议在施工图设计阶段按发包人确认资料进行校核……”
- “涉及外部接口条件的内容，应按合同约定程序确认后落实……”

与审查 Skill 的衔接：

- 写作 Skill 每次输出后附带内部来源记录；
- 审查 Skill 先查硬风险，再查表达质量；
- 审查问题按“必须修改 / 建议修改 / 需人工确认”输出；
- 写作 Skill 只根据审查结果做局部修正，不擅自扩大章节。

## 10. BIDFORGE 审查 Skill 草案

Skill 名称：`bidforge-bid-review`

目标：  
审查模型生成或人工修改的标书内容，识别废标风险、扣分风险、事实错误、服务范围越界和表达质量问题。

输入材料：

- 待审查章节；
- 招标文件事实库；
- 评分点矩阵；
- 图纸证据表；
- 禁止表达清单；
- 章节类型。

输出格式：

| 问题编号 | 所在章节 | 问题类型 | 问题描述 | 依据来源 | 风险等级 | 是否可能废标 | 修改建议 |
|---|---|---|---|---|---|---|---|

审查流程：

1. 识别章节任务和对应评分点。
2. 进行项目事实一致性检查。
3. 进行图纸依据检查。
4. 进行招标范围和服务边界检查。
5. 进行过度承诺和绝对化表达检查。
6. 进行外项目痕迹检查。
7. 进行空泛套话和标题正文匹配检查。
8. 输出风险表。
9. 给出可执行修改建议。

风险分类：

- 高风险：可能影响响应性、形成事实错误、扩大承诺、违反招标文件。
- 中风险：可能影响得分、专业可信度、表达稳妥性。
- 低风险：表达可优化、重复、结构不够清晰。

废标风险判断：

- 与投标内容、工期、质量标准、权利义务、承包人建议等响应性要求冲突；
- 引入错误项目事实；
- 服务承诺明显超出或背离招标文件；
- 关键格式或组成缺失。

扣分风险判断：

- 未覆盖评分点；
- 重难点分析不针对；
- 解决措施不详实；
- 项目理解停留在套话；
- 设计构思没有结合项目实际。

图纸无依据内容判断：

- 检测具体参数、系统配置、空间组织、流线、结构形式等是否能找到 A/B 级来源；
- 无来源则标记为高风险，建议删除或改为复核性表达。

空泛表达判断：

- 段落中只有宏观目标，无项目事实、专业对象、措施、成果；
- 同类形容词连续堆叠；
- 多段句式高度相似。

过度承诺判断：

- 识别“确保、保证、全部、彻底、零风险、100%”；
- 涉及审查通过、造价、工期、安全、质量的绝对结果承诺一律预警。

无关项目内容判断：

- 与当前项目事实库不一致的地名、功能、项目名称、单位、道路名、建筑类型均标为高风险。

## 11. BIDFORGE Auditor / 规则审计器设计

Auditor 的作用：在规则进入正式 Skill 前进行审查，防止 BIDFORGE 越学越偏。

### Auditor 审查清单

| 审查项 | 判断标准 |
|---|---|
| 是否有来源 | 规则必须来自招标文件、图纸、参考片段、反面问题或人工修改差异 |
| 是否可执行 | 规则能被写作 Skill 或审查 Skill 使用，不能只是口号 |
| 是否可审查 | 审查器能通过关键词、事实库、证据等级或结构矩阵检测 |
| 是否误把特例当通用 | 规则不能包含项目名称、地点、具体面积等特例，除非标记为项目规则 |
| 是否学入坏写法 | 从参考标书提取的规则必须通过过度承诺和空泛检查 |
| 是否硬编码反面教材 | 负面规则应抽象化，不得把反面教材项目事实写成禁止规则 |
| 是否过度保守 | 规则不能导致系统什么都不敢写；D 级证据仍可写原则 |
| 是否过度激进 | 规则不能允许系统在无来源时具体化 |
| 是否与招标文件优先级冲突 | 冲突时不得进入 Production Skill |
| 是否有回滚方案 | 新规则必须先进入 Candidate Skill，经小样本测试后再发布 |

### 规则进入正式 Skill 前需要满足的条件

1. 有明确来源；
2. 可描述触发条件；
3. 有推荐处理方式；
4. 不与招标文件优先原则冲突；
5. 经至少一次小章节回放测试；
6. 人工确认后再进入 Production Skill。

### Candidate Skill 与 Production Skill 的区别

- Candidate Skill：候选规则区，用于实验、回放和人工验证；不会直接影响前台正式生成。
- Production Skill：正式规则区，只包含经过审计、测试、人工确认的规则；可用于前台生产系统。

不能让新规则直接影响前台生产系统的原因：

- 新规则可能来自单个项目特例；
- 参考标书里也可能有坏表达；
- 反面教材里的项目事实可能被误学；
- 未测试规则可能让系统过度保守或乱承诺；
- 前台生产需要稳定，规则进化需要隔离。

## 12. 首次小章节回放测试建议

建议首测章节：`合理化建议` 中的“小范围版本”，不要测试整章。

推荐范围：

- `1.6.1 合理化建议总体说明`
- `1.6.2 关于总体规划与功能组织的建议`

为什么适合第一轮测试：

- 招标文件明确承包人建议书评分点，且分值虽小但很能测试“项目针对性”。
- 反面教材 `02_合理化建议.md` 已经暴露过度展开、内容过散、图纸依据不足等问题。
- 该章节既需要体现主动性，又必须避免改变建设范围、降低标准或扩大服务责任。

需要输入材料：

- 招标文件事实库；
- 承包人建议书评分标准；
- 发包人要求中的工程规模、功能、设计范围；
- 反面教材片段；
- 参考表达库；
- 若有，补充当前项目总平面、初设或方案图纸。

预期输出：

- 一版短而稳的合理化建议小节；
- 内部来源记录；
- 不确定事项清单；
- 审查 Skill 风险报告；
- 人工修改差异记录。

审查重点：

- 是否响应区位、交通、用地现状、周边配套、重难点和解决措施；
- 是否使用了招标文件事实；
- 是否凭空写了图纸细节；
- 是否扩大服务范围；
- 是否过度承诺；
- 是否比反面教材更短、更清楚、更有边界。

成功标准：

- 没有外项目痕迹；
- 没有无来源具体参数；
- 每个建议都有“依据、动作、成果、边界”；
- 审查 Skill 高风险为 0；
- 人工修改主要集中在风格和偏好，而不是事实错误和合规风险。
