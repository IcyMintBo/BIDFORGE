# BIDFORGE 通用规则体系与样本归纳报告

生成日期：2026-05-28  
分析范围：`I:\CodexProjects\BIDFORGE\samples`  
报告定位：为 BIDFORGE 建立跨项目复用的规则体系、样本归纳方法、Skill 演化机制。  
重要边界：本报告不建立任何单一项目事实库；样本中的具体项目事实仅作为案例，不进入通用 Skill。

## 1. 本阶段目标重新定义

本阶段不是训练模型，不是针对某一个项目生成标书，也不是建立某一个项目的事实库。

本阶段真正要完成的是：从 `samples` 中的多项目、多类型材料里，抽象出 BIDFORGE 以后面对任意标书项目时都可以复用的方法。也就是说，系统要先学会“怎么读资料、怎么区分证据、怎么提取表达、怎么识别风险、怎么沉淀规则”，而不是先学会某一个项目的名称、面积、地点或工期。

本阶段的核心产物应是：

- 样本分类规则；
- 招标文件信息抽取规则；
- 项目事实识别规则；
- 项目事实与通用规则的隔离机制；
- 参考标书表达模式提取规则；
- 反面教材问题归纳规则；
- 通用写作规则；
- 通用审查规则；
- Candidate Skill / Production Skill 的规则沉淀机制；
- Auditor 规则审计机制。

具体项目事实可以作为测试数据和案例材料使用，但不得直接写入 Production Skill。比如某项目的名称、地点、建筑面积、工期、专业范围，只能存在于“项目事实库”或“测试样例”中，不能变成 BIDFORGE 的通用规则。

多项目识别规则：

1. 文件名、封面、第一页、目录、页眉页脚中出现的项目名称，优先作为所属项目线索。
2. 如果正文出现多个项目名，应标记为“可能混入外项目内容”，不能直接作为正样本。
3. 参考标书只能学习表达模式，不学习项目事实。
4. 反面教材只能提取问题类型，不把整篇内容硬编码为禁止文本。
5. 招标文件可以作为信息抽取样本，但其中具体事实只属于该项目。

## 3. 招标文件通用信息抽取规则

BIDFORGE 面对任意招标文件时，都应先抽取信息类型，而不是先写正文。信息抽取结果进入当前项目的“项目事实库”和“约束库”，不进入通用 Skill。

| 信息类型 | 为什么要抽取 | 通常出现位置 | 写作时如何使用 | 审查时如何使用 | 是否高优先级约束 |
|---|---|---|---|---|---|
| 项目名称 | 防止项目名错误和外项目痕迹 | 招标公告、投标人须知、合同协议书、发包人要求 | 所有章节统一口径 | 检查全文项目名一致性 | 是 |
| 招标人 / 建设单位 | 确定服务对象和称谓 | 招标公告、合同条款 | 写作中使用“招标人/发包人/建设单位”时保持一致 | 检查是否误写其他业主 | 是 |
| 建设地点 | 支撑区位、交通、周边条件分析 | 招标公告、项目概况、发包人要求 | 用于项目理解、区位分析 | 检查是否混入外地名 | 是 |
| 项目规模 | 确定项目体量和复杂度 | 项目概况、发包人要求、工程量清单 | 用于工程概述和重难点判断 | 检查面积、数量、规模是否无来源 | 是 |
| 建设内容 | 明确写作对象 | 招标公告、发包人要求、设计任务书 | 决定章节重点和专业范围 | 检查是否写入无关工程 | 是 |
| 招标范围 | 确定投标响应边界 | 招标公告、投标人须知、合同条款 | 控制正文不越界 | 检查服务范围扩大 | 是 |
| 服务范围 | 明确投标人承诺的服务内容 | 合同条款、发包人要求、设计任务书 | 写后续服务、现场配合、设计变更时使用 | 检查是否承诺额外服务 | 是 |
| 设计范围 | 区分设计工作内容 | 发包人要求、合同专用条款 | 写设计大纲、设计组织、成果交付 | 检查是否漏写专项设计或写超范围 | 是 |
| 施工范围 | 区分施工组织内容 | 招标范围、合同条款、工程范围 | 写施工组织方案时使用 | 检查设计章节是否混入施工管理 | 是 |
| 专业范围 | 确定专业章节覆盖 | 发包人要求、设计任务书、技术标准 | 组织建筑、结构、给排水、电气等章节 | 检查专业遗漏或无关专业混入 | 是 |
| 工期 / 服务周期 | 控制进度承诺 | 招标公告、投标人须知、合同条款、发包人要求 | 写进度计划和服务安排 | 检查工期是否错误或过度承诺 | 是 |
| 质量标准 | 控制质量承诺 | 招标公告、合同条款、发包人要求 | 写质量目标和质量保证措施 | 检查是否低于招标要求 | 是 |
| 投标文件组成 | 防止漏项 | 投标人须知、投标文件格式 | 决定文件目录和章节顺序 | 检查缺章缺表风险 | 是 |
| 评分办法 | 决定内容取舍和章节权重 | 评标办法、评分标准 | 章节结构应映射评分点 | 检查得分点覆盖 | 是 |
| 废标 / 否决条款 | 防止硬伤 | 评标办法、投标人须知、否决性条款 | 写作时避开否决风险 | 高风险审查清单 | 是 |
| 格式要求 | 保证文件形式合规 | 投标文件格式、电子投标要求 | 控制标题、签章、页码、表格 | 检查格式响应 | 是 |
| 发包人要求 | 明确最低技术和服务要求 | 第五章、设计任务书、技术规范 | 写技术响应和设计方案 | 检查是否低于最低要求 | 是 |
| 合同边界 | 控制责任边界 | 合同条款、专用合同条款 | 避免扩大服务、责任和承诺 | 检查履约风险 | 是 |
| 成果交付要求 | 明确设计成果和数量 | 发包人要求、合同条款 | 写成果交付计划 | 检查是否漏成果 | 是 |
| 后续服务要求 | 明确配合范围 | 发包人要求、合同条款 | 写设计交底、变更、审查回复、驻场等 | 检查是否少写或多承诺 | 是 |
| 限额设计要求 | 控制造价表达 | 发包人要求、合同条款 | 写限额设计、技术经济比选 | 检查是否把限额设计写成降低标准 | 是 |
| 特殊专项要求 | 捕捉项目特有重点 | 发包人要求、专项说明、技术规范 | 写专项设计章节 | 检查专项遗漏或凭空添加 | 是 |

样例说明：某样本招标文件中出现了产业园、工期、专项设计、限额设计等内容，这些只能作为“招标文件信息类型抽取示例”，不进入通用规则。通用规则只能写成：“遇到任意招标文件时，应抽取项目名称、招标范围、评分办法、发包人要求等字段，并标注来源。”

## 4. 项目事实与通用规则的隔离机制

BIDFORGE 必须把“事实”和“规则”分开，否则系统会把某个项目的具体情况误学成长期偏见。

| 内容类别 | 定义 | 示例 | 建议存放位置 | 是否允许进入 Production Skill |
|---|---|---|---|---|
| 项目事实 | 仅对当前项目成立的信息 | 某项目名称、地点、面积、工期、建设内容、招标人 | 当前项目 `project_facts` / 工作区数据库 / 项目 Markdown | 否 |
| 通用规则 | 对多数标书项目都成立的方法或约束 | 没有来源不得写具体面积；评分点必须映射章节结构 | `Production Skill` 的规则区 | 是，经 Auditor 通过后 |
| 项目类型规则 | 对某类项目常见但非所有项目通用的关注点 | 产业园关注物流/厂房/市政接口；学校关注安全疏散/改扩建扰民；市政道路关注交通组织/排水 | `project_type_rules`，按项目类型标签管理 | 可以，但必须带类型限定 |
| 表达模式 | 可复用的句式、语气和组织方式 | 审慎表达、设计服务边界表达、评分点响应表达 | `expression_library` | 可以，但不能包含具体项目事实 |
| 禁止模式 | 高风险写法或错误模式 | 过度承诺、外项目痕迹、施工口径混入设计服务、无图纸依据具体化 | `review_rules` / `forbidden_patterns` | 是，经 Auditor 通过后 |
| 测试样例 | 用来验证规则的具体片段 | 模型原稿、人工修改稿、反面教材片段 | `lab_cases` / Forge Lab 数据集 | 否 |
| 规则候选 | 从样本中提炼但未审计的规则 | “合理化建议应先写边界再写建议” | `Candidate Skill` | 否，审计后才可晋级 |

隔离流程：

1. 读取样本时，先给每条信息打标签：`fact`、`rule`、`type_rule`、`expression`、`forbidden_pattern`、`case`。
2. 任何包含项目名称、地名、面积、工期、单位名称、设备参数的内容，默认进入 `fact` 或 `case`，不得直接进入 `rule`。
3. 如果一条规则需要项目类型前提，必须写明适用类型，例如“厂房类项目”“学校改扩建项目”“市政道路项目”。
4. Candidate Skill 中的规则不能直接影响前台生成。
5. Production Skill 只能接收经过 Auditor 审核、去项目化、可执行、可审查的规则。

## 5. 参考标书的通用表达提取规则

参考标书的价值不在于“内容正确”，而在于可以抽出成熟标书的表达方式、章节组织方式和专业口径。BIDFORGE 不应学习参考标书中的项目事实。

可提取内容：

| 可提取对象 | 提取方法 | 产出形式 |
|---|---|---|
| 章节组织方式 | 提取目录层级和章节任务，不复制标题中的项目特例 | 章节模板，如“项目理解-重难点-措施-服务保障” |
| 设计院口径 | 提取“设计依据、设计输入、专业提资、校审、成果交付、后续服务”等表达 | 设计服务表达库 |
| 审慎表达 | 提取“结合资料复核、按程序确认、在满足要求前提下”等语气 | 风险安全句式库 |
| 评分点响应方式 | 观察优秀文本如何把评分点拆成小节 | 评分点映射模板 |
| 质量进度服务表达 | 提取质量体系、进度节点、设计交底、图纸会审、审查回复等结构 | 质量进度表达模式 |
| 多专业协同表达 | 提取建筑、结构、机电、景观、市政、消防等接口协同方式 | 专业协同句式库 |
| 限额设计表达 | 提取“不降低功能、安全、质量前提下控制造价”的表达 | 限额设计安全表达库 |
| 后续服务表达 | 提取设计变更、现场技术服务、技术交底、审查配合的边界表达 | 后续服务边界库 |
| 风险控制表达 | 提取问题清单、复核机制、闭环管理的写法 | 风险控制表达库 |

不能迁移内容：

- 项目名称；
- 地名；
- 建设规模；
- 功能定位；
- 业主名称；
- 道路名称；
- 设备参数；
- 地区政策；
- 旧规范；
- 项目特有结论；
- 企业资质、体系认证、获奖情况；
- 参考标书中的“保证 100%”“确保无风险”等过度承诺。

提取规则：

1. 先删除所有项目专有名词，再判断剩余句式是否仍有价值。
2. 如果去掉项目事实后句子仍能表达方法、边界或流程，可进入表达库。
3. 如果句子依赖具体项目参数才能成立，只能作为案例。
4. 如果参考标书存在过度承诺、旧规范、空泛口号，也应进入负面规则库。

## 6. 反面教材的通用负面规则提取方法

反面教材不是“整篇都错”，而是里面包含可以被抽象的错误模式。BIDFORGE 要学的是“为什么这类写法危险”，不是简单记住某段文字不好。

| 问题类型 | 识别特征 | 为什么危险 | 审查 Skill 如何检测 | 写作 Skill 如何避免 |
|---|---|---|---|---|
| 泛泛而谈 | 连续出现宏观词，如统筹、提升、优化、协同，但没有项目事实、评分点或成果 | 看起来长，实际不响应项目 | 检测段落是否缺少事实库字段、评分点、专业对象 | 每段至少绑定一个事实、评分点、专业对象或成果 |
| 套话堆叠 | 多段重复“有利于、提高、减少、保障、形成闭环” | 降低信息密度，像 AI 扩写 | 句式重复率和抽象词密度检测 | 用“依据-动作-成果-边界”替代堆词 |
| 章节过度展开 | 一个小节扩展出过多三级/四级标题 | 主次失衡，评委难抓重点 | 统计标题数量和评分点数量是否失衡 | 章节围绕评分点和必要专业展开 |
| 无项目事实支撑 | 没有项目名称、建设内容、范围或资料来源 | 不像针对当前项目 | 段落与项目事实库无匹配 | 写前先调取事实库，无事实则写原则 |
| 无评分点对应 | 内容很多但不对应评分标准 | 可能低分 | 建立评分点覆盖矩阵 | 先搭评分点结构，再写正文 |
| 无图纸依据具体化 | 未见图纸却写流线、容量、层高、道路、消防细节 | 可能事实错误 | 具体参数/系统词必须有 A/B 级来源 | 无证据只写复核建议 |
| 过度承诺 | 确保、保证、彻底、零风险、全部解决 | 形成履约风险 | 绝对化词库扫描 | 改为审慎表达和过程控制 |
| 服务范围扩大 | 写成长期驻场、管理施工、监督施工、负责运营 | 扩大合同责任 | 服务边界词扫描 | 只写招标文件明确服务 |
| 施工管理口径混入设计服务 | 设计章节写班组、机械、现场施工组织 | 角色混乱 | 按章节类型检测施工词 | 设计章节只写设计成果和技术配合 |
| 外项目内容迁移 | 出现外地名、外项目名、外业主、外工程类型 | 硬伤，可能直接失信 | 与当前项目事实库反向比对 | 参考标书只调用表达模式 |
| 专业术语堆砌 | BIM、智慧、低碳、全生命周期等无落点 | 显得高级但空 | 术语后无动作和成果则预警 | 术语必须连接对象、方法、成果 |
| 绝对化表达 | 所有、任何、完全、最优、唯一、100% | 不稳妥，容易被抓 | 绝对词检测 | 除招标文件硬要求外改为条件表达 |
| 前后矛盾 | 一处说不越界，一处写超范围服务 | 合同边界不一致 | 跨段比对服务、工期、质量、面积等 | 使用统一口径表 |

负面规则提取流程：

1. 从反面教材中截取问题片段。
2. 标注问题类型。
3. 删除项目事实，保留抽象问题。
4. 写出检测条件。
5. 写出替代写法原则。
6. 放入 Candidate Skill。
7. 经 Auditor 审核后，才可进入 Production Skill。

## 7. 通用图纸/证据分级规则

证据分级是 BIDFORGE 的核心安全机制。它决定系统能把话说到什么程度。

| 等级 | 来源 | 可以写到什么程度 | 推荐表达 | 处理方式 |
|---|---|---|---|---|
| A 级 | 招标文件明确要求、发包人要求、合同条款、正式澄清补遗 | 可以写成确定事实和硬约束 | “招标文件明确要求……” | 最高优先级，写作和审查都必须引用 |
| B 级 | 当前项目图纸、方案、初设、地勘、市政接口资料、经确认技术资料 | 可以写具体技术内容，但应绑定来源 | “根据图纸/方案所示……” | 可具体化，但需记录来源 |
| C 级 | 从 A/B 级资料合理推断，但资料未直接写明 | 可以写方向、建议、复核动作，不写定论 | “宜结合……进一步复核” | 审慎表达，列入复核项 |
| D 级 | 行业通用原则、设计常识、经验方法 | 只能写原则、流程、控制方法 | “应按现行规范和项目资料进行校核” | 不得写具体参数 |
| E 级 | 无依据、来自外项目、来自想象、来源不明 | 禁止写成事实 | 不写，或写“待资料确认” | 删除或转人工确认 |

必须有 A/B 级证据才能写的内容：

- 项目名称、地点、规模、面积、层数、工期；
- 招标范围、服务范围、设计范围、施工范围；
- 专项设计内容；
- 具体道路宽度、管线接口、消防组织、设备容量；
- 结构体系、基础形式、荷载参数；
- 材料品牌、设备型号、工艺参数；
- 具体规范条文和地方政策要求；
- 甲方特殊承诺、驻场服务、成果份数。

C/D 级内容的审慎表达方式：

- “建议结合当前项目图纸进一步复核……”
- “可在满足招标文件及现行规范要求的前提下……”
- “宜作为施工图阶段重点校核内容……”
- “涉及外部接口条件的，应按发包人确认资料落实……”

E 级内容处理：

- 不进入正文；
- 标记为“无依据”；
- 如果用户坚持使用，要求补充来源或人工确认；
- 审查 Skill 将其列为高风险。

## 8. 正式标书文风与措辞规则

### 8.1 本章目标

BIDFORGE 生成的内容必须像正式投标文件正文，而不是像 AI 的写作说明、章节提纲、审查意见或系统自我防御说明。

正式标书正文应直接进入项目理解、设计判断、技术控制、服务边界、成果落实、程序衔接和风险复核，不应暴露“系统正在怎么写”“模型为什么不敢写”“后续章节准备写什么”等内部过程。

### 8.2 禁止“章节说明感”

正式正文中不得出现明显的章节导读式表达，例如：

- 本节按照……展开；
- 本章主要介绍……；
- 以下从……方面说明；
- 后续章节将……；
- 本部分拟……。

这类表达更像写作说明或目录导读，不像投标文件正文。BIDFORGE 写作 Skill 应将其转换为项目理解、设计判断或工作控制逻辑。

| 类型 | 错误写法 | 推荐改写方向 |
| --- | --- | --- |
| 章节说明感 | 本节按照“工程概述、功能定位、建设条件、设计理念”四个方面展开。 | 对本项目的理解，需从工程组成、功能运行、建设条件和设计方向四个层面展开。工程组成明确建设范围，功能运行体现项目使用逻辑，建设条件识别外部接口和实施约束，设计方向则承接后续深化、优化和保障措施。 |

可执行规则：

- 如果句子主语是“本节、本章、本部分、以下、下文、后续章节”，且谓语是“介绍、说明、展开、阐述、分析”，应判定为章节说明感。
- 章节说明感不能直接进入正式稿，应改为该内容对项目设计、服务实施或成果控制的作用。
- 如果确实需要交代逻辑，应使用“项目理解需从……把握”“设计工作应以……为基础”等正式正文表达。

### 8.3 禁止“写作任务感”

正式正文中不得写成“提醒作者应该怎么写”，例如：

- 工程概述部分应……；
- 本章节需要……；
- 此处应体现……；
- 本段应说明……。

这类表达是在指导写作，不是标书正文。应改成项目事实、设计判断或控制要求。

| 类型 | 错误写法 | 推荐改写方向 |
| --- | --- | --- |
| 写作任务感 | 工程概述部分应准确交代项目基本事实。 | 项目基本事实是后续设计判断的基础。项目名称、建设地点、建设单位、建设规模、单体组成和建设内容需要保持统一，避免后续章节在基础信息上出现偏差。 |

可执行规则：

- 如果正文中出现“某部分应写、某章节需要、此处应体现、本段应说明”等句式，应判定为写作任务感。
- 写作任务感应转换为“项目事实为何重要”“设计如何控制”“成果如何保持一致”。
- Skill 生成正文前，可以保留内部写作任务，但正式输出必须删除这类内部语言。

### 8.4 禁止“自我防御感”

内部风控逻辑不能直接出现在投标正文中。以下表达适合做系统规则，不适合做标书正文：

- 坚持正式稳妥的投标文件口径；
- 不作脱离资料条件的绝对化承诺；
- 避免废标风险；
- 防止扩大责任；
- 不能写没有依据的内容。

这些意思应转换为专业、稳妥、自然的标书表达。

| 类型 | 错误写法 | 推荐改写方向 |
| --- | --- | --- |
| 自我防御感 | 对项目的理解坚持正式、稳妥、可执行的投标文件口径，不作脱离资料条件和自身控制范围的绝对化承诺。 | 对已明确的建设规模、单体组成、功能定位和实施边界，后续设计将保持一致并深化落实；对需发包人确认、主管部门审查或外部单位核实的内容，设计工作将结合项目程序做好技术衔接，保证成果表达稳妥、边界清晰。 |

可执行规则：

- “避免废标、不能写、无依据不能、风险口径、正式稳妥口径、防止责任扩大”等词组不得原样进入正文。
- 如需表达边界，应转化为“以已明确资料为基础”“结合发包人确认及主管部门审查要求”“做好技术衔接”“保持成果边界清晰”。
- 审查 Skill 发现自我防御感时，应标为“正文口吻问题 / AI 写作痕迹问题”，通常属于必须修改。

### 8.5 控制机械重复

标书正文允许稳健，但不能机械。BIDFORGE 应检查并压制以下重复：

- 连续多个句子以“本项目……”开头；
- 连续重复“应围绕、应结合、应通过、进一步、有效、全面、系统”等词；
- 同一段中多次出现“应围绕”或类似结构；
- 每节结尾都使用相同收束句；
- 标题换了但正文句式没有变化。

可执行规则：

- 同一自然段内，同类句式不得连续重复超过 2 次。
- 同一小节内，收束句必须推进语义，不能只是重复标题。
- “应、需、可、建议、通过、围绕、结合、落实”等高频词需要交替使用。
- 如果连续两段结构高度相似，应重写其中一段，使其分别承担“事实判断、技术控制、实施衔接、成果落地”等不同功能。

审查方式：

- 统计段落内句首结构，如果连续 3 句以上相同，应提示改写。
- 统计高频控制词，如果单段出现同一控制词超过 3 次，应提示替换或重组句子。
- 对相邻段落进行结构比对，如果均为“本项目 + 应 + 通过 + 目标”的模板句，应判定为机械套写。

### 8.6 清理 AI 痕迹和排版残留

生成后必须检查并清理以下内容：

- 错位引号；
- 多余空格；
- 孤立标点；
- Markdown 残留；
- 半截短语；
- 复制粘贴残留；
- 中英文标点混乱；
- 类似“功能 ‘ ’ 多”“设计前 ‘ ’ 端”的异常片段。

可执行规则：

- 凡出现异常符号组合、半截词组、明显排版残留，必须优先修复，不得进入正式稿。
- 正式稿中不得保留 Markdown 标记，例如 `#`、`**`、无意义列表符号、代码块符号。
- 中英文标点混用应统一为中文正文习惯，英文缩写、标准号和文件名除外。
- 审查 Skill 应将排版残留列为“必须修改”，即使其不构成废标风险，也不能进入交付稿。

### 8.7 禁止“后续章节将……”式内部说明

正式标书不是论文导读，不应写：

- 后续章节将围绕……展开；
- 下文将进一步说明……；
- 后续设计将在后续章节中……。

这类表达应改为该判断对设计、实施和成果的支撑作用。

| 类型 | 错误写法 | 推荐改写方向 |
| --- | --- | --- |
| 后续章节导读 | 后续设计将在后续章节中围绕该功能定位展开设计内容、合理化建议和优化措施。 | 该功能定位将作为后续设计内容、合理化建议和优化措施的共同基础，使总体判断、施工图成果和实施保障保持一致。 |

可执行规则：

- “后续章节、下文、以下内容、后文将”一类内部导读词，应从正式正文中清除。
- 如果确实需要承接，应写成“该判断将作为……基础”“该要求将贯穿……过程”“相关成果应在……中保持一致”。

### 8.8 从“怎么写”转为“项目怎么理解、怎么控制、怎么落实”

BIDFORGE 写作时应避免暴露内部写作逻辑。正文中不应出现以下内部语言：

- 本节、本章、本文；
- 写作、口径、表达；
- 资料不足不能写；
- 需要体现、应该说明；
- 避免废标风险、防止扩大责任。

正式正文应转化为以下语义类型：

| 内部写作逻辑 | 正式正文转化方向 |
| --- | --- |
| 怎么组织章节 | 项目理解、设计判断 |
| 为什么不能乱写 | 服务边界、成果依据 |
| 如何避免风险 | 程序衔接、技术复核 |
| 如何响应评分点 | 质量控制、进度控制、专业协同 |
| 资料不足 | 需进一步确认、结合后续审查、在深化阶段复核 |

### 8.9 进入 Skill 的规则要求

该文风规则应同时进入四个机制：

- 写作 Skill：生成前先建立写作边界，生成后执行正式正文口吻转换。
- 审查 Skill：新增“正文口吻问题 / AI 写作痕迹问题”。
- 规则提取 Skill：从人工改稿中提取“AI 说明语变成标书正文”的差异规则。
- Auditor：审查任何候选规则是否会把内部说明、风控提示或写作任务暴露到前台正文。

## 9. BIDFORGE 通用写作 Skill 草案

Skill 名称：`bidforge-bid-writing`

目标：  
在任意项目中，根据当前项目事实库、评分点矩阵、证据表、表达库和正式标书文风规则，生成稳妥、合规、可审查，并且像正式投标文件正文的标书章节。

输入材料：

- 当前项目事实库；
- 招标文件信息抽取表；
- 评分点矩阵；
- 图纸/方案/初设/地勘证据表；
- 章节任务；
- 项目类型标签；
- 参考表达库；
- 禁止模式库；
- 正式标书文风与措辞规则；
- 用户指定风格或口径。

工作流程：

1. 识别章节类型：承包人建议书、设计大纲、实施方案、质量进度服务、风险审查等。
2. 读取当前项目事实库，只使用当前项目事实，不从参考标书拿事实。
3. 读取评分点矩阵，确定本章必须覆盖的内容。
4. 读取证据分级表，判断哪些内容可以具体写，哪些只能审慎写。
5. 调用表达库，选择适合的句式和章节组织方式。
6. 生成章节前先输出“写作边界”：可写内容、不可写内容、需复核内容。
7. 生成正文时控制服务边界，不扩大承诺。
8. 将内部写作边界转换为正式正文，不暴露“怎么写、为什么不能写、后续章节将写什么”等系统过程。
9. 检查句式重复、章节说明感、写作任务感、自我防御感和 AI 痕迹。
10. 生成后自动交给审查 Skill。

必须遵守：

- 招标文件优先；
- 当前项目事实优先；
- 评分点结构优先；
- A/B 级证据才可具体化；
- C/D 级内容必须审慎表达；
- E 级内容禁止写成事实；
- 不迁移参考标书项目内容；
- 不把施工管理口径写进设计服务章节；
- 不使用过度承诺和绝对化表达；
- 不在正式正文中出现“本节按照、本章主要介绍、以下从、后续章节将、本部分拟”等章节说明感表达；
- 不在正式正文中出现“本章节需要、此处应体现、本段应说明”等写作任务感表达；
- 不把“避免废标风险、不能写没有依据的内容、防止扩大责任”等内部风控语言写入正文；
- 同一自然段内同类句式不得连续重复超过 2 次；
- 输出前必须清理 Markdown 残留、错位引号、孤立标点、多余空格、半截短语和异常符号组合。

章节前写作边界模板：

该模板只作为内部控制表或 Forge Lab 过程记录，不得直接进入前台正式标书正文。

| 类型 | 内容 |
|---|---|
| 本章可使用事实 | 从当前项目事实库读取 |
| 本章必须响应评分点 | 从评分点矩阵读取 |
| 本章可具体化内容 | A/B 级证据支持 |
| 本章需审慎表达内容 | C/D 级证据 |
| 本章禁止写入内容 | E 级、外项目事实、超范围承诺 |

与审查 Skill 衔接：

- 写作 Skill 输出后，不直接进入正式稿；
- 审查 Skill 先检查硬风险，再检查扣分风险，再检查正文口吻和表达质量；
- 修改时只按审查结果局部调整，不自行扩大章节范围。

## 10. BIDFORGE 通用审查 Skill 草案

Skill 名称：`bidforge-bid-review`

目标：  
对任意项目标书内容进行合规性、响应性、事实来源、图纸依据、服务边界、正文口吻和表达质量审查。

输入材料：

- 待审查文本；
- 当前项目事实库；
- 招标文件信息抽取表；
- 评分点矩阵；
- 证据分级表；
- 项目类型标签；
- 禁止模式库；
- 正式标书文风与措辞规则；
- 章节类型。

审查项目：

| 审查项 | 检查方式 | 输出风险 |
|---|---|---|
| 招标响应 | 文本是否覆盖招标范围、发包人要求、评分点 | 废标风险 / 扣分风险 |
| 事实来源 | 项目事实是否存在于当前事实库 | 高风险 |
| 评分点覆盖 | 评分点矩阵是否有空缺 | 中/高风险 |
| 服务范围扩大 | 是否写了招标文件未要求的服务、责任、驻场、管理 | 高风险 |
| 过度承诺 | 是否出现绝对化结果承诺 | 中/高风险 |
| 外项目痕迹 | 是否出现其他项目名、地名、业主、功能 | 高风险 |
| 图纸无依据内容 | 技术细节是否缺少 A/B 级证据 | 高风险 |
| 空泛套话 | 是否无事实、无评分点、无成果 | 中风险 |
| 施工口径混入设计 | 设计章节是否写施工组织管理 | 中/高风险 |
| 前后矛盾 | 工期、范围、质量、服务、事实是否冲突 | 高风险 |
| 正文口吻 / AI 写作痕迹 | 是否出现章节说明感、写作任务感、自我防御感、后续章节导读、Markdown 残留或机械重复 | 中/高风险 |

风险输出：

| 风险等级 | 含义 | 处理方式 |
|---|---|---|
| 高 | 可能影响响应性、造成事实错误、扩大合同责任或出现外项目硬伤 | 必须修改 |
| 中 | 可能影响得分、专业可信度或表达稳妥性 | 建议修改 |
| 低 | 主要是语言、结构、重复、清晰度问题 | 可优化 |

废标风险、扣分风险、表达优化的区分：

- 废标风险：违反投标文件组成、格式、响应性、工期、质量、权利义务、否决条款等硬要求。
- 扣分风险：评分点未覆盖、重难点不准确、解决方案不具体、项目理解泛化。
- 表达优化：句式重复、语气不稳、标题和正文不完全匹配、文本不够顺。
- 正文口吻问题：内容像写作说明、审查意见、AI 自我防御或内部流程提示，不像正式投标文件正文。

审查结果表建议：

| 问题编号 | 所在章节 | 问题类型 | 问题描述 | 依据来源 | 风险等级 | 风险类别 | 修改建议 |
|---|---|---|---|---|---|---|---|

正文口吻问题 / AI 写作痕迹问题专项输出：

| 原句 | 问题类型 | 为什么不像正式标书 | 推荐改写 | 是否属于必须修改 |
|---|---|---|---|---|
| 本节按照……展开 | 章节说明感 | 像目录导读或写作说明，没有直接形成项目判断 | 改为“项目理解需从……把握，并服务于……成果落实” | 是 |
| 工程概述部分应…… | 写作任务感 | 在指导作者写作，而不是投标正文 | 改为“项目基本事实是后续设计判断的基础……” | 是 |
| 避免废标风险、不能写没有依据的内容 | 自我防御感 | 暴露内部风控逻辑，语气不适合正式正文 | 改为“对需确认内容将结合项目程序做好技术衔接……” | 是 |
| 后续章节将围绕……展开 | 后续章节导读 | 像报告导语，不构成当前章节实质内容 | 改为“该判断将作为……共同基础” | 通常是 |
| `**重点**`、孤立标点、错位引号 | AI 痕迹 / 排版残留 | 存在 Markdown 或排版异常，影响正式交付质量 | 清除残留并恢复正式中文标点 | 是 |

必须修改判定：

- 涉及章节说明感、写作任务感、自我防御感的句子，原则上必须修改。
- Markdown 残留、异常符号、半截短语必须修改。
- 轻微句式重复可列为建议修改；如果连续影响一个小节的正式感，应升级为必须修改。

## 11. BIDFORGE 规则提取 Skill 草案

Skill 名称：`bidforge-rule-extraction`

目标：  
从多个项目、多类样本、多轮“模型原稿 vs 人工修改稿”中提取可复用规则，并将其送入 Candidate Skill，待 Auditor 审查。

输入材料：

- 招标文件；
- 参考标书；
- 反面教材；
- 模型原始输出；
- 人工修改稿；
- 用户批注；
- 审查报告；
- 项目类型标签。

输出内容：

- 样本分类结果；
- 项目事实提取结果；
- 通用规则候选；
- 项目类型规则候选；
- 表达模式候选；
- 禁止模式候选；
- 正文口吻规则候选；
- 人工修改差异归纳；
- Candidate Skill 补丁建议。

如何从样本中抽象规则：

1. 先识别样本所属项目和材料类型。
2. 删除项目特有事实，观察剩余内容是否仍有通用价值。
3. 将“可复用方法”写成触发条件和处理动作。
4. 将“坏写法”写成检测特征和替代策略。
5. 将“AI 说明语、写作任务语、内部风控语”与人工改稿中的正式正文进行对照，提取正文口吻规则。
6. 将规则送入 Candidate Skill。

如何判断规则是否可泛化：

- 不依赖具体项目名称、地点、面积、工期；
- 可用于多个项目类型；
- 能写成明确触发条件；
- 能被审查 Skill 检测；
- 不与招标文件优先原则冲突；
- 不把内部提示、审查意见或写作说明带入正式正文；
- 经至少一个回放案例验证。

如何判断只是项目特例：

- 包含具体地名、业主、面积、道路、设备、功能；
- 只对某个项目类型成立；
- 离开该项目后无法判断是否适用；
- 需要图纸或发包人确认才能成立。

模型原稿 vs 人工修改稿差异记录：

| 字段 | 说明 |
|---|---|
| 原稿片段 | 模型生成内容 |
| 人工修改片段 | 用户认可后的内容 |
| 修改类型 | 删除、降风险、补事实、改口径、压缩、重组 |
| 问题标签 | 泛化、越界、无依据、过度承诺等 |
| 正文口吻变化 | 是否从章节说明、写作任务、自我防御、AI 痕迹改为正式标书正文 |
| 是否可泛化 | 是 / 否 / 需更多样本 |
| 规则候选 | 从差异中提炼出的规则 |
| 进入位置 | Candidate Skill / 表达库 / 禁止模式库 / 项目事实库 |

正文口吻规则提取方式：

- 如果人工修改删除了“本节、本章、以下、后续章节将”等导读词，应提取为“章节说明感禁止规则”。
- 如果人工修改将“本章节需要……”改为项目判断，应提取为“写作任务感转换规则”。
- 如果人工修改将“避免废标、不能乱写、无依据不能写”等内部风控语改为服务边界表达，应提取为“自我防御感转换规则”。
- 如果人工修改压缩了连续相同句式，应提取为“机械重复控制规则”。
- 如果人工修改清除了 Markdown、错位引号、半截短语，应提取为“AI 痕迹清理规则”。

正文口吻规则不能直接等同于某个用户喜欢的句子。只有当它能写成“触发特征 + 判断理由 + 改写方向 + 审查方法”时，才允许进入 Candidate Skill。

## 12. BIDFORGE Auditor 通用审计机制

Auditor 是 Forge Lab 的规则审计器。它不是写标书，也不是审查标书正文，而是审查“新规则能不能进入正式系统”。

Auditor 审查重点：

| 审查项 | 审查问题 | 处理结果 |
|---|---|---|
| 项目事实误学 | 是否把项目名称、面积、地点、工期当成通用规则 | 拒绝进入 Production Skill |
| 参考标书坏表达 | 是否把过度承诺、旧规范、企业宣传套话学进去 | 退回 Candidate Skill |
| 反面教材硬编码 | 是否把反面教材具体内容直接写成禁止规则 | 要求抽象为问题类型 |
| 可执行性 | 规则是否能指导写作动作 | 不可执行则退回 |
| 可审查性 | 审查 Skill 是否能检测该规则 | 不可审查则改写 |
| 过度保守 | 是否导致系统什么都不敢写 | 降级为审慎规则 |
| 过度激进 | 是否允许无依据具体化或乱承诺 | 拒绝 |
| 项目类型限定 | 是否只适用于产业园、学校、市政道路等类型 | 加项目类型标签 |
| 正文口吻污染 | 是否把“本节说明、此处应写、避免废标、不能乱写”等内部语言固化为正文表达 | 拒绝进入 Production Skill，改为内部规则 |
| AI 痕迹可检测性 | 是否能检测 Markdown 残留、异常符号、半截短语、机械重复等成文质量问题 | 不可检测则补充触发特征 |
| 文风规则可改写性 | 是否不仅指出“不像标书”，还给出正式正文改写方向 | 无改写方向则退回 |
| 进入 Production 条件 | 是否有来源、测试、人工确认 | 满足才允许进入 |

Candidate Skill 与 Production Skill：

- Candidate Skill：规则试验区。允许保留未完全验证的候选规则，用于回放测试、人工审查和版本比较。
- Production Skill：正式规则区。只包含去项目化、可执行、可审查、经过人工确认的规则。

规则晋级流程：

1. 规则提取 Skill 生成候选规则。
2. Auditor 检查来源、适用范围、风险和可执行性。
3. 小章节回放测试验证规则效果，包括事实合规、招标响应和正式正文口吻。
4. 用户或人工审查确认。
5. 进入 Production Skill。
6. 前台写作和审查才能调用该规则。

为什么不能让新规则直接影响前台：

- 样本可能来自不同项目；
- 参考标书不一定都是好表达；
- 反面教材中也可能有可用句式；
- 单个项目事实容易污染通用能力；
- 未审计规则可能让系统过度保守或过度激进；
- 未审计文风规则可能让前台正文出现 AI 说明感、审查意见感或内部风控提示。

## 13. 第一轮最小闭环建议

第一轮不要直接进入具体项目标书写作，应先做一个通用能力闭环。

### 1. 多项目样本分类

目标：识别每份样本属于哪个项目、哪类材料、能学什么、不能学什么。  
输出：`sample_inventory`。

### 2. 通用信息类型抽取

目标：从招标文件中抽取“信息类型”，不是固化某项目事实。  
输出：`tender_extraction_schema`。

### 3. 正向表达模式抽象

目标：从参考标书中提取章节组织、设计院口径、审慎表达、服务边界表达。  
输出：`expression_library_candidate`。

### 4. 正式正文口吻规则抽象

目标：从参考标书、反面教材和人工改稿中提取“什么像正式标书正文，什么像 AI 写作说明或审查意见”。  
输出：`formal_bid_style_candidate`。

重点区分：

- 通用文风规则：如禁止章节说明感、写作任务感、自我防御感，可以进入 Candidate Skill。
- 具体改写句子：只能作为示例，不直接进入 Production Skill。
- 用户个人偏好：需经过多轮样本验证，不能一次性固化为通用规则。

### 5. 负面问题类型抽象

目标：从反面教材中提取问题类型，而不是整篇负样本。  
输出：`negative_pattern_candidate`。

### 6. 生成通用 Skill 草案

目标：形成写作 Skill、审查 Skill、规则提取 Skill、Auditor 的候选版本。  
输出：`Candidate Skill v0.1`。

### 7. 选择一个小章节做验证

建议选择“合理化建议”或“设计质量进度及后续服务保证措施”作为小章节验证对象。验证时必须明确：

| 内容 | 是否可进入 Production Skill |
|---|---|
| 通用规则，如“无依据不得写具体参数” | 可以，经 Auditor 审查 |
| 项目事实，如某项目面积、地点、工期 | 不可以，只进项目事实库 |
| 项目类型规则，如产业园关注物流组织 | 可以，但必须带项目类型标签 |
| 表达模式，如审慎表达、边界表达 | 可以，去项目化后进入表达库 |
| 正文口吻规则，如禁止“本节按照……展开” | 可以，经回放验证后进入文风规则库 |
| 负面样例原文 | 不可以，只保留问题标签和检测规则 |

第一轮成功标准：

- 样本不再被错误合并为一个项目；
- 通用规则中不出现具体项目名称、面积、地点、工期；
- 写作 Skill 能先读取当前项目事实库，再写章节；
- 审查 Skill 能识别外项目痕迹、无依据具体化、过度承诺；
- 审查 Skill 能识别章节说明感、写作任务感、自我防御感、AI 痕迹和机械重复；
- Auditor 能阻止项目事实进入 Production Skill；
- 小章节回放能清楚区分“规则、事实、项目类型规则、表达模式、文风规则、禁止模式”。

本阶段最终目标不是让 BIDFORGE 立刻写完整标书，而是让它先拥有一套干净的学习方法：看样本不乱学，写内容不乱编，正文不像 AI 说明书，审查风险有依据，规则升级有门槛。
