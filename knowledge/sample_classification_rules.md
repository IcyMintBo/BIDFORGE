# BIDFORGE 样本分类规则

用途：把 `samples/` 目录中的材料整理为可管理的多项目样本库，防止把不同项目、不同材料类型混在一起。

本文件服务于路线图第 4 阶段：样本入库与分类。

## 分类优先级

1. 先判断所属项目。
2. 再判断材料类型。
3. 再判断是否适合作为通用规则来源。
4. 最后判断哪些内容必须禁止迁移。

如果无法确认所属项目，应标记为 `未知/需人工确认`，不得强行归并到已有项目。

## 项目识别规则

| 识别项 | 判断方法 | 处理方式 |
|---|---|---|
| 项目名称 | 从封面、招标公告、标题、页眉、目录、正文高频项目名中识别 | 作为项目分组依据 |
| 招标人/业主 | 从招标公告、封面、投标文件封面识别 | 作为辅助分组依据 |
| 建设地点 | 从项目概况、工程基本情况识别 | 作为辅助分组依据 |
| 项目编号/招标编号 | 从封面、公告、投标文件编号识别 | 用于区分同名或近似项目 |
| 文件路径 | `招标文件`、`参考标书`、`反面教材` 等文件夹只作为初步线索 | 不得仅凭文件夹名确认项目 |
| 文本内容冲突 | 文件名与正文项目名不一致 | 以正文封面/公告为准，并标记异常 |

## 材料类型

| 类型 | 定义 | 可学习内容 | 禁止迁移内容 |
|---|---|---|---|
| 招标文件 | 当前或历史项目招标文件、发包人要求、评标办法 | 信息抽取字段、约束识别、评分点识别、否决条款识别 | 具体项目事实不得进入通用 Skill |
| 参考标书 | 历史投标文件、技术标、承包人建议书、设计方案文本 | 章节组织、正式文风、审慎表达、服务边界表达、评分响应方式 | 项目名称、地点、业主、规模、设备参数、旧规范、项目结论 |
| 反面教材 | 模型原稿、用户不满意稿、风险稿、过度扩写稿 | 问题类型、触发特征、审查方法、改写方向 | 原文不得整段当负样本；具体项目事实不得硬编码为规则 |
| 方案资料 | 当前或历史项目方案、初设、专项设计资料 | 资料证据分级、图纸/方案约束识别、专业表达边界 | 不得当成通用项目事实 |
| 图纸资料 | CAD、PDF 图纸、总平、专业图 | B 级证据识别方法、图纸关联写作规则 | 无关项目图纸事实不得迁移 |
| 模型原稿 | AI 生成的未审查内容 | 识别常见生成问题 | 不得当成可学习正文 |
| 人工改稿 | 用户认可或修改后的内容 | 差异学习、文风规则、边界控制 | 单句偏好不得直接成为通用规则 |
| 审查报告 | 人工或 Skill 输出的风险审查 | 审查规则、风险分类、改写建议 | 具体项目事实不得固化 |

## 可进入通用能力的内容

- 信息类型抽取方法。
- 证据等级判断方法。
- 章节组织方法。
- 审慎表达模式。
- 服务边界表达模式。
- 评分点响应方法。
- 负面问题类型。
- 正文口吻规则。
- 审查触发特征。

## 不得进入通用能力的内容

- 项目名称、地点、招标人、业主。
- 建设面积、投资额、工期、建设内容、单体组成。
- 当前项目评分分值、条款编号原文。
- 道路名称、设备型号、材料品牌、系统参数。
- 地方政策、旧规范、企业联系人、联系方式。
- 参考标书中的项目特有判断。
- 反面教材中的具体项目内容。

## 入库字段

每个样本至少记录：

| 字段 | 说明 |
|---|---|
| 样本编号 | `S-类型-序号` |
| 文件路径 | 相对 `samples/` 的路径 |
| 所属项目 | 无法确认则写 `未知/需人工确认` |
| 项目编号 | 内部项目分组编号，如 `P001` |
| 材料类型 | 招标文件 / 参考标书 / 反面教材 / 方案资料 / 图纸资料 / 其他 |
| 是否可作为通用规则来源 | 是 / 有条件 / 否 |
| 可抽象出的规则类型 | 信息抽取 / 表达模式 / 负面模式 / 文风规则 / 审查规则 / 测试样例 |
| 禁止迁移内容 | 明确写出不能迁移的项目事实 |
| 后续处理 | 入库 / 待人工确认 / 仅测试 / 暂不使用 |

## 进入后续流程的条件

- 招标文件样本可用于完善 `tender_extraction_schema`，但具体事实只进入案例。
- 参考标书样本可用于提取表达模式，必须去项目化。
- 反面教材可用于提取负面模式，不能整篇作为负样本。
- 任何候选规则进入 Production 前必须经过小章节回放和 Auditor。
